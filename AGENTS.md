# TubeDigest Agent Rules

TubeDigest is an issue-driven multi-agent MVP. Agents must work from documentation, not guesses.

## Source of truth
Before starting any task, read the relevant docs:

1. `docs/project.md`
2. `docs/architecture.md`
3. `docs/pipeline.md`
4. `docs/contracts.md`
5. `docs/testing.md`
6. `docs/engineering.md`
7. `docs/issue-guidelines.md`
8. `docs/roadmap.md`
9. `docs/decisions.md`
10. `docs/glossary.md`

If documentation and implementation disagree, stop and report the inconsistency.

## Global rules

- Work must be driven by a GitHub Issue.
- One issue = one primary owner = one PR.
- Do not expand scope without creating a follow-up issue.
- Every issue must include tests.
- Never commit secrets or generated media artifacts.
- Never call real YouTube, RabbitMQ, ffmpeg, or Groq from unit tests.
- Validate contracts with Pydantic on Python side and Zod on frontend side.
- Keep external integrations behind adapters.
- Keep worker handlers testable without RabbitMQ.

## Agent roles

- `architect-agent`: creates and checks issue decomposition from docs.
- `backend-agent`: owns FastAPI, SQLAlchemy, SQLite, API, backend tests.
- `frontend-agent`: owns React, Vite, TypeScript, Tailwind, UI tests.
- `contracts-agent`: owns Pydantic/Zod schemas and contract tests.
- `worker-agent`: owns ytd, ffmpeg, transcribe, summary, tags workers.
- `infra-agent`: owns Docker, RabbitMQ setup, Makefile, CI infrastructure.
- `github-agent`: owns GitHub CLI flow, issue/PR hygiene, labels, workflows.
- `test-engineer`: owns test design, test additions, CI failure diagnosis.
- `reviewer`: reviews PRs for scope, tests, contracts, maintainability.

## Definition of Done
A task is done only when:

- implementation matches the issue scope;
- tests are added or updated;
- contract changes are reflected in docs/tests;
- CI-relevant commands pass locally where possible;
- PR includes evidence of tests run;
- docs are updated if behavior changed.

## Worktree rules

- Use git worktrees for parallel agent work.
- Do not run multiple agents in the same checkout.
- One worktree = one agent stream = one branch family.
- Agents may read any file in the repository, including `AGENTS.md` and `docs/`.
- Agents may modify only files required by the assigned issue.
- Agents must not switch branches unless explicitly asked by the user.

Recommended worktree setup is generated by:

```powershell
.\scripts\create-worktrees.ps1
```

## Pull Request policy

After completing an assigned issue, the agent must:

1. verify the issue acceptance criteria;
2. run relevant tests;
3. commit changes with a conventional commit message;
4. push the current branch;
5. create a GitHub Pull Request with `gh pr create`;
6. link the PR to the issue using `Closes #<issue-number>`.

The agent must not merge its own PR.
